# 이중화 환경에서 Redis 구성 시 고려사항

웹 서비스 이중화 환경에서 로그인 세션을 유지하기 위해 Redis 구성을 검토하면서 겪은 내용을 정리합니다.

---

## 문제/상황

현재 구성하려는 환경은 다음과 같습니다.

- 웹 서버 이중화 (App 서버 2대)
- NCP Load Balancer를 통한 트래픽 분산
- 로그인 이후 쿠키 기반 세션 유지
- Web / API 서버가 동일 인스턴스에서 동작

이 구조에서 다음과 같은 상황이 발생할 수 있습니다.

```
사용자 → LB → 서버1 (로그인)
사용자 → LB → 서버2 (API 호출)
```

즉, 로그인과 이후 요청이 서로 다른 서버에서 처리될 수 있습니다.

초기에는 단일 서버(운영 APP 서버)에 Redis를 설치하여 세션을 저장하는 방안을 고려했습니다.  
하지만 이 방식이 이중화 환경에 적합한지에 대해 검토가 필요했습니다.

---

## 해결 방법 / 개요

이 글에서는 아래 내용을 중심으로 정리합니다.

- Redis의 기본 개념
- 이중화 환경에서 Redis 사용 시 주의사항
- 설치형 Redis와 Managed Redis 비교
- 실제 구성 변경 과정

---

## 아키텍처 / 흐름

### 기존 고려 구조

```
User → LB → App1 / App2 → Redis (단일 서버)
```

### 변경 후 구조

```
User → LB → App1 / App2 → NCP Cloud DB for Cache (Redis)
```

---

## 사전 준비

- NCP 환경 (Load Balancer, VM)
- Redis 사용 경험 (세션 저장 용도)
- 웹 서비스 이중화 구조

---

## 1. Redis 개념 (간략)

Redis는 메모리 기반 Key-Value 저장소로, 빠른 조회 속도가 필요한 데이터 처리에 사용됩니다.

주로 다음과 같은 용도로 사용됩니다.

- 로그인 세션 저장
- 캐시 데이터 저장
- 간단한 상태 정보 관리

특히 로그인 세션의 경우, 다음과 같은 흐름으로 사용됩니다.

```
로그인 → 세션 생성 → Redis 저장
요청 → 세션 조회 → 인증 처리
```

---

## 2. 이중화 환경에서 Redis 주의사항

### 2-1) 왜 공용 Redis가 필요한가

이중화 환경에서는 요청이 항상 같은 서버로 가지 않습니다.

```
로그인 → 서버1
API 요청 → 서버2
```

이 경우 각 서버에 Redis가 따로 있다면 세션을 공유할 수 없습니다.

따라서 반드시 **모든 서버가 동일한 Redis를 사용해야 합니다.**

---

### 2-2) 단일 Redis 서버의 문제점

공용 Redis를 사용하더라도, 이를 단일 서버에 구성할 경우 문제가 발생합니다.

#### 1) 단일 장애 지점 (SPOF)

- Redis 서버 장애 시 전체 로그인 세션 사용 불가
- 모든 API 요청이 인증 실패로 이어짐

#### 2) 운영 중 재시작 영향

- Redis 재시작 시 세션 데이터 유실 가능
- 전체 사용자 로그아웃 발생

#### 3) 이중화 구조와 불일치

웹 서버는 이중화되어 있지만  
세션 저장소는 단일 서버라면 구조적으로 불균형 상태입니다.

---

## 3. Managed Redis vs 설치형 Redis

### 3-1) 설치형 Redis (단일 서버)

- 서버에 직접 Redis 설치
- 초기 구성은 간단
- 장애 대응 및 운영을 직접 수행해야 함

### 3-2) Managed Redis (Cloud DB for Cache)

- 클라우드에서 제공하는 Redis 서비스
- 이중화 및 장애 대응 구조 포함
- 운영 부담 감소

---

### 비교 정리

| 항목 | 설치형 Redis | Managed Redis |
|------|-------------|--------------|
| 세션 공유 | 가능 | 가능 |
| 장애 대응 | 수동 대응 | 자동 대응 |
| 이중화 | 별도 구성 필요 | 기본 제공 |
| 운영 부담 | 높음 | 낮음 |

---

## 4. 실제 상황 및 해결 과정

초기에는 운영 APP 서버에 Redis를 설치하여  
세션 저장소로 사용하는 방안을 고려했습니다.

하지만 아래 이유로 해당 방식은 적절하지 않다고 판단했습니다.

- 이중화 환경에서 단일 Redis는 장애 시 전체 서비스 영향 발생
- 세션은 인증과 직결되므로 안정성이 중요
- OpenSearch, DB 등 주요 구성 요소는 이미 이중화 또는 Managed 서비스 사용

결론적으로, 세션 저장소 역시 동일 기준으로 구성하는 것이 필요하다고 판단했습니다.

따라서 다음과 같이 구조를 변경했습니다.

- Redis → NCP Cloud DB for Cache 사용
- DB → RDS 사용
- 주요 상태 저장소 → Managed 서비스 기반으로 통일

---

## 정리

이중화 환경에서 Redis는 단순 캐시가 아니라  
**로그인 상태를 유지하는 핵심 구성 요소**입니다.

따라서 다음 기준으로 접근하는 것이 적절합니다.

- 모든 서버가 공통으로 접근 가능한 구조
- 단일 장애 지점 제거
- 운영 중단 없이 유지 가능한 구성

이러한 조건을 만족하기 위해  
Redis는 설치형보다 **Managed 서비스 기반으로 구성하는 것이 적합합니다.**
